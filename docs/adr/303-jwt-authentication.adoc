=== ADR-303: OAuth 2.0 / JWT Authentication
* **Status**: Accepted
* **Category**: Service Implementation Details
* **Context**: The service requires secure access control. We need to verify the identity of the caller and ensure they have appropriate permissions (Roles).
* **Decision**: Use JSON Web Tokens (JWT) for authentication.
* **Details**:
** **Domain/Issuer**: `org.javafreedom.kdiab`
** **Audience**: `profile` (Configurable)
** **Roles**: Mapped to strict Enum values (`PATIENT`, `DOCTOR`, `ADMIN`).
*** `PATIENT` (`user` claim): Standard access to own resources.
*** `DOCTOR`: Read-only access to specific patients (Delegated).
*** `ADMIN`: Full superuser access to all resources.
** **Delegated Access**: The token may contain claims that grant read-only access to specific resources, respecting external relationships.
*** `allowed_patients`: A list of UUIDs representing the patients this user (e.g., Doctor) is allowed to access.
*** Example Claim: ` "allowed_patients": ["123e4567-e89b-12d3-a456-426614174000"] `
* **Rationale**:
** **Standard**: JWT is the industry standard for stateless authentication in microservices.
** **Interoperability**: Easily integrates with Identity Providers (Keycloak, Auth0, etc.).
** **Stateless**: No session state required on the server.

* **Addendum (OpenID Connect & JWKS)**:
** **Decision**: Migrate from symmetric `HMAC256` token validation to asymmetric `RS256` using Keycloak's JSON Web Key Set (JWKS).
** **Details**: The backend now dynamically fetches the Identity Provider's public keys via a URL (e.g., `http://keycloak:8080/realms/kdiab-profiles/protocol/openid-connect/certs`).
** **Rationale**:
*** Allows Keycloak to rotate keys securely without backend downtime.
*** Prevents the need to hardcode shared symmetric secrets `jwt.secret` across environments.
*** Standardized approach for verifying third-party OAuth 2.0 / OIDC provider signatures.
