= ADR-112: CI/CD Performance Optimizations
:toc: left
:icons: font

* **Status**: Accepted
* **Date**: 2026-04-02
* **Context**: 
As the codebase grows, the GitHub Action workflows for Backend and Frontend, alongside the Semantic Release process, have become slower due to redundant build steps and lack of precise trigger path filtering. Specifically, the Backend Docker build was re-compiling the entire project from scratch, and the Release workflow was re-running the full build and test suites already completed in the Continuous Integration phase.

* **Decision**: 
1. **Precise Path Filtering**: We implement strict Git `paths` filters in `backend-ci.yml` and `frontend-ci.yml` to ensure that specific CI logic only runs when relevant directories or critical root configuration files (e.g., `build.gradle.kts`, `package.json`) are modified.
2. **Docker Build Acceleration**: We optimize the `Dockerfile` to leverage the pre-built JAR artifacts generated by the GitHub Action runner's Gradle environment. This eliminates the need for a multi-stage compilation within Docker during CI, significantly reducing image build time.
3. **Artifact Sharing and Reuse**: Instead of re-building artifacts in subsequent workflows, the `backend-ci.yml` now uploads the built Docker image as a temporary GitHub Action artifact. The `release.yml` workflow (triggered by `workflow_run`) downloads and reuses this artifact, bypassing redundant compilation and verification.
4. **Job Parallelization**: We separate long-running or optional tasks like SonarCloud into parallel steps or jobs where feasible, reducing the time-to-feedback for developers.

* **Rationale**:
** **Efficiency**: Drastically reduces CI/CD runtime, leading to faster developer feedback and quicker deployment cycles.
** **Cost Reduction**: Minimizes GitHub Action runner minute consumption.
** **Consistency**: Guarantees that the exact same artifact verified by the CI suite is the one tagged and released by `semantic-release`.

* **Consequences**:
** The `backend/Dockerfile` now requires a pre-built JAR to be present in the build context. For local development, users must run `./gradlew backend:build` before running `docker build`.
** Changes to the project's root build logic now trigger both frontend and backend CIs to ensure global consistency.
** Artifact storage usage in GitHub Actions increases slightly during the overlap between CI and Release runs.
